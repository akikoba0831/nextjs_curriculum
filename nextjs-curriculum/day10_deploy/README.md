## 📚 解説

### `next build` は何を作っているのか
`npm run dev`は開発用に毎回コードをその場でコンパイルしますが、本番公開の前には`next build`でアプリを最適化された形に事前ビルドします。ここでは大まかに以下が行われます。

- 静的にできるページ(Day4で学んだキャッシュされる`fetch`を使うページなど)は事前にHTMLとして生成される
- 動的なページ(リクエストごとに変わるページ)はサーバーで実行する関数としてまとめられる
- Client Componentのコードは最小限のJavaScriptバンドルとしてまとめられる(Day3で学んだ通り、Server Componentのコードはここに含まれない)

### なぜVercelは「ゼロコンフィグ」で動くのか
VercelはNext.jsを開発している会社が運営するホスティングサービスで、`next build`が生成する成果物(静的アセット・サーバー関数・キャッシュ設定など)をそのまま解釈できるように作られています。そのため、他の汎用的なホスティングサービスのように「サーバーの起動方法」や「ルーティングの設定」を自分で書く必要がなく、GitHubリポジトリを接続するだけで動きます。これは裏を返すと、Vercel以外の環境(自前のサーバーなど)にデプロイする場合は、Next.js側でこれらを代替する設定が必要になる、ということでもあります。

### 環境変数: ビルド時とリクエスト時の違い
環境変数(APIキーなど)には2種類の扱いがあります。

- **サーバー側でのみ使う変数**(例: DBの接続文字列): サーバーの実行時にのみ読み込まれ、ブラウザには一切送られない
- **`NEXT_PUBLIC_`接頭辞を付けた変数**: ビルド時にクライアント向けのJavaScriptに**埋め込まれ**、ブラウザからも参照できるようになる

`NEXT_PUBLIC_`を付け忘れるとクライアント側の値が`undefined`になり、逆に秘密情報にうっかり`NEXT_PUBLIC_`を付けてしまうとブラウザに漏洩する、という事故が起きやすいので注意してください。

### プレビューデプロイ
Vercelは、GitHub上でPull Requestを作るたびに、そのブランチ専用の一意なURL(プレビューデプロイ)を自動生成します。`main`ブランチにマージする前に、実際に公開されるのと同じ環境で動作確認できるのが特徴です。チーム開発では「コードレビューと同時に実際の画面も確認できる」という点で重宝されます。

## ✅ 実施タスク

### 1. GitHubにプッシュする

```bash
git init
git add .
git commit -m "first commit"
```

GitHub上で新しいリポジトリを作成し、指示されたコマンドでプッシュしてください。

### 2. Vercelでデプロイする

1. https://vercel.com にアクセスし、GitHubアカウントでログイン
2. 「Add New Project」から、先ほど作ったリポジトリを選択
3. 設定はそのままで「Deploy」をクリック

数分待つと、`https://your-app.vercel.app` のようなURLでアプリが公開されます。

### 3. コードを更新して自動デプロイを確認する

適当な文言を変更してGitHubにpushしてみてください。
Vercelが自動で再デプロイしてくれるのを確認しましょう。
